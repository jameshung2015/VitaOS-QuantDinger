# 前后端仅保留中英语言改造复盘（2026-05-12）

## 1. 背景与目标
- 现象：语言切换可触发状态变化，但页面文案存在“部分不跟随切换”的问题。
- 目标：前后端统一仅支持 `en-US` 与 `zh-CN`，移除其他语言入口与分支，避免运行时加载错误语言 chunk。

## 2. 根因总结
- 前端曾存在多语言入口与动态加载能力，导致运行环境可能请求历史语言包。
- 后端存在多语分支（启发式文案与语言映射），与“仅中英”目标不一致。
- 部署侧最关键问题：前端容器未挂载本地 `frontend/dist`，而是使用镜像内静态资源。
  - 结果：即使本地 dist 已更新，线上页面仍可能继续加载旧 `app/chunk` 文件。
  - 表现：`ChunkLoadError`（请求旧 hash 的 `lang-zh-CN.*.js`）。

## 3. 关键修复动作

### 3.1 前端（源码）
- 语言选择器只保留 `en-US` 与 `zh-CN`。
- `store`、`bootstrap`、`locales` 增加语言白名单归一化，非法值回退到默认语言。
- 动态语言加载增加构建限制：仅允许打包 `en-US`/`zh-CN` 语言文件。

### 3.2 后端
- 语言标准化仅支持 `en-US` 与 `zh-CN`。
- 回测 AI 分析相关映射、提示词、启发式兜底逻辑改为仅中英。
- 清理 `backtest` 路由中的多语言分支，确保后端行为与前端一致。

### 3.3 构建与部署
- 前端重新构建并同步 dist。
- 重新构建并重启 `quantdinger-frontend` 容器，使容器内静态资源与最新 dist 对齐。
- 验证运行页面实际加载脚本 hash 已切到新构建产物。

## 4. 验证结果
- 页面语言菜单仅 2 项：English、简体中文。
- 语言切换后，`localStorage` 中 `lang`/`app_language` 与界面文案同步变化。
- 后端关键 Python 文件语法编译通过。

## 5. 本次经验教训
- 经验 1：仅改源码不等于线上生效，必须确认“容器实际 serving 的静态资源来源”。
- 经验 2：语言问题需按“入口 -> 状态 -> 动态加载 -> 后端归一化 -> 部署资源”全链路排查。
- 经验 3：凡是带 hash 的静态资源改造，部署后应优先核验页面实际加载的 `app/chunk` 文件名。
- 经验 4：若容器无 volume 挂载，更新 dist 后必须重建镜像/容器，而非仅重启。

## 6. 建议固化项
- 在发布检查清单新增：
  - 检查语言菜单项数量与白名单一致。
  - 检查浏览器实际加载的 `app*.js/chunk-vendors*.js` hash 是否为当前构建。
  - 检查后端语言白名单与前端白名单一致。
- 在 CI 增加简单扫描：阻止新增 `zh-TW/ja-JP/ko-KR/...` 语言分支进入主干（如果策略是仅中英）。

## 7. 结论
本次“仅中英语言”改造已完成代码与运行验证。后续若扩展语言，建议先定义统一白名单策略，再同步更新前端入口、构建规则、后端归一化与部署流程。
